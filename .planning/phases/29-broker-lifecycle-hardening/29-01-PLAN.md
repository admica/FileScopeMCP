**Change 2 (BRKR-02): Add crash handlers**

Per D-06, add both `uncaughtException` and `unhandledRejection` handlers. Register them at module level BEFORE the `main().catch(...)` block (i.e., between line 122 and line 126). This ensures they catch errors during all execution phases including startup.

Add this block after the closing brace of `main()` (after line 122) and before `main().catch(...)`:

```typescript
// ─── Crash handlers (BRKR-02, D-06) ─────────────────────────────────────────

process.on('uncaughtException', (err: Error) => {
  log(`Broker crash (uncaughtException): ${err.message}\n${err.stack ?? ''}`);
  fs.rmSync(SOCK_PATH, { force: true });
  fs.rmSync(PID_PATH, { force: true });
  process.exit(1);
});

process.on('unhandledRejection', (reason: unknown) => {
  const msg = reason instanceof Error ? reason.message : String(reason);
  log(`Broker crash (unhandledRejection): ${msg}`);
  fs.rmSync(SOCK_PATH, { force: true });
  fs.rmSync(PID_PATH, { force: true });
  process.exit(1);
});
```

Key details:
- Handlers are synchronous (no await) per Node.js docs — process state is undefined after uncaught exception.
- `{ force: true }` makes cleanup safe even if PID/socket files do not yet exist (ENOENT silently ignored).
- The existing `shutdownStarted` guard in `shutdown()` prevents double-execution during graceful shutdown. Crash handlers bypass this guard intentionally — `{ force: true }` makes double-cleanup safe.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PID file on disk | Any local process can write to ~/.filescope/broker.pid — attacker could inject a PID to prevent broker startup |
| Socket file on disk | Any local user could create a fake broker.sock to intercept connections |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-01 | Tampering | PID file (broker.pid) | accept | PID file is user-owned in ~/.filescope/; local attacker with write access already has full control. No PII at risk. |
| T-29-02 | Denial of Service | checkPidGuard false positive | mitigate | D-05 dual check (PID alive AND socket exists) prevents stale PID from blocking broker startup |
| T-29-03 | Denial of Service | Crash without cleanup | mitigate | D-06 uncaughtException/unhandledRejection handlers clean up socket+PID before exit(1) |
| T-29-04 | Denial of Service | Unbounded drain wait | mitigate | D-01 drain timeout (15s default) prevents indefinite hang on stuck job during shutdown |
</threat_model>
