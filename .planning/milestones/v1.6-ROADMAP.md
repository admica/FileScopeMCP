# Milestone v1.6: Symbol-Level Intelligence

**Status:** ✅ SHIPPED 2026-04-23
**Phases:** 33-35
**Total Plans:** 10
**Total Tasks:** 10

## Overview

Elevated FileScopeMCP from file-granular to symbol-granular for three daily-use LLM queries — killed grep for symbol navigation, exposed import-names on dependent edges, and added "changed since" re-orientation. TS/JS only; other languages deferred pending adoption signal.

## Phases

### Phase 33: Symbol Extraction Foundation

**Goal**: TS/JS files populate a `symbols` table during scan with top-level declarations (name, kind, line range, export flag), and the dep parser records imported names and import lines during the same AST walk — no second parser pass per file.
**Depends on**: Phase 32 (v1.5 milestone complete)
**Requirements**: SYM-01..08, IMP-01..03, PERF-01 (12 total)
**Plans**: 5 plans

Plans:

- [x] 33-01: Medium-repo fixture + bench-scan CLI + PERF-01 baseline capture
- [x] 33-02: Schema migration — symbols + kv_state tables, imported_names + import_line columns
- [x] 33-03: AST parser single-pass emits symbols + ImportMeta alongside edges
- [x] 33-04: Repository functions (upsertSymbols, getSymbolsByName, getSymbolsForFile, deleteSymbolsForFile, setEdgesAndSymbols) + kv_state helpers
- [x] 33-05: Migration-time bulk-symbol-extract gated by kv_state flag + inspect-symbols CLI

**Success Criteria (all TRUE):**

1. `symbols` table with indexes on `(name)` and `(path)`; additive migration clean on pre-v1.6 DBs
2. Single AST walk emits both edges and symbols — no second `parse()` call
3. Six kinds (function, class, interface, type, enum, const) with line ranges + `isExport` flag
4. `file_dependencies` carries `importedNames` + `importLines`; namespace imports record `*`
5. First startup bulk-extracts symbols for every tracked TS/JS file (kv_state gate)
6. `npm run inspect-symbols <path>` CLI emits symbols for debugging
7. Re-exports (`export * from './foo'`) do NOT populate the re-exporting file
8. Pre-implementation baseline captured (`baseline.json` at commit 860fe61: 1833ms self-scan / 332ms medium-repo)

### Phase 34: Symbol-Aware MCP Surface

**Goal**: LLM agents resolve a symbol to `{path, line, kind}` in a single MCP call via `find_symbol`, and `get_file_summary` surfaces per-file exports plus dependent-edge import-names.
**Depends on**: Phase 33
**Requirements**: FIND-01..05, SUM-01..04 (9 total)
**Plans**: 2 plans

Plans:

- [x] 34-01: Repository helpers — findSymbols (GLOB prefix + dual COUNT/LIMIT) + getDependentsWithImports (Map aggregation with NULL coerce)
- [x] 34-02: Register find_symbol MCP tool + enrich get_file_summary with exports[] + rich dependents[]; 14-tool registry + contract tests

**Success Criteria (all TRUE):**

1. `find_symbol(name)` returns `{path, name, kind, startLine, endLine, isExport}` — exact case-sensitive, prefix via trailing `*`
2. `exportedOnly=true` by default; private helpers only with explicit false
3. Standardized `{items, total, truncated?: true}` envelope; no-match → `{items: [], total: 0}` (not error)
4. `get_file_summary.exports: [{name, kind, startLine, endLine}]` populated for TS/JS
5. `get_file_summary.dependents[]` upgraded from `string[]` to `[{path, importedNames, importLines}]`
6. Non-TS/JS files return `exports: []` and `dependents[].importedNames: []` without error

### Phase 35: Changed-Since Tool + Watcher Integration

**Goal**: Agents re-orient after multi-edit runs with one MCP call; watcher re-extracts symbols on file change via the existing single-pass AST walk, keeping `symbols` as fresh as `file_dependencies`.
**Depends on**: Phase 33, Phase 34
**Requirements**: CHG-01..05, WTC-01..03, PERF-02 (9 total)
**Plans**: 3 plans

Plans:

- [x] 35-01: Repository helpers (getFilesChangedSince, getFilesByPaths) + extend deleteFile with transactional symbols cascade (WTC-02)
- [x] 35-02: Register list_changed_since MCP tool with dual-mode dispatch (ISO-8601 + git SHA); extend ErrorCode union
- [x] 35-03: Watcher lifecycle regression guard + PERF-02 bench-scan capture with baseline restoration

**Success Criteria (all TRUE):**

1. `list_changed_since(timestamp)` returns `[{path, mtime}]` for files whose mtime > timestamp; ISO-8601 accepted
2. `list_changed_since(sha)` runs `git diff --name-only <sha> HEAD` and intersects with DB
3. Invalid `since` returns `INVALID_SINCE`; SHA mode without `.git` returns `NOT_GIT_REPO`
4. Tool response lists only files currently in DB — no deletion tombstones
5. FileWatcher change events re-extract symbols alongside edges (same throttle, single-pass walk)
6. FileWatcher unlink invokes `deleteSymbolsForFile(path)` via transactional cascade
7. End-of-milestone scan wall-time regression < 15% vs Phase 33 baseline (soft); hard-fail at 25%
   - Actual: self-scan +13.75% (1833→2085ms), medium-repo +9.64% (332→364ms) — both within soft threshold

## Milestone Summary

**Decimal Phases:** None.

**Key Decisions:**

- File-granular → symbol-granular is the one structural change for v1.6; all other LLM-surface gaps are workarounds for this mismatch — ruthlessly cut from 8 candidates to 3 high-value tools
- TS/JS only v1.6; Python/Go/Ruby deferred to v1.7 pending adoption signal
- Parser emits symbols in single AST pass alongside edges — no second walk (PERF-critical mandate)
- Additive schema only (`symbols` + `kv_state` tables, `imported_names` + `import_line` columns) — no breaking changes to existing tool responses
- Deletion tracking deferred — `list_changed_since` returns only existing files with mtime
- Inlined `getDependentsWithImports` return type (single call site, no new interface)
- Used direct `insertDepRow()` helper in tests rather than extending `setEdges()` — needed fine-grained control over NULL imported_names + package_import rows
- Established GLOB+bracket-escape as the case-sensitive prefix-match pattern (new to codebase, no PRAGMA needed)
- Extended `tests/unit/tool-outputs.test.ts` as the contract-test home (avoided a single-file `tests/contract/` directory)
- `find_symbol` description authored as `string[].join(' ')` so the length probe can regex-extract without JS eval
- Skipped `/gsd-verify-work` on phases 33 + 35 — verification added retroactively at milestone close (see [v1.6-MILESTONE-AUDIT.md](./v1.6-MILESTONE-AUDIT.md))

**Issues Resolved:**

- LLM agents had no symbol-granularity surface — required `grep` fallback and reading raw files to locate a definition
- `get_file_summary.dependents[]` was `string[]` — agents couldn't tell what names were imported from each source without re-reading files
- No "changed since T" re-orientation — agents lost context after multi-edit runs
- Watcher had no symbols-path — file edits left `symbols` stale after initial bulk extract

**Issues Deferred:**

- Python / Go / Ruby symbol extraction — wait for v1.6 adoption signal
- Method-level symbols on classes — reachable via class start/end line ranges
- Call-site resolution (`who calls foo`) — needs full type registry, HIGH complexity
- Deletion tombstones in `list_changed_since` — not needed elsewhere yet
- Fuzzy symbol search — exact + prefix deemed sufficient
- Symbol importance scoring — file-level importance is already approximate
- Nexus UI rendering of symbols — v1.6 is MCP-surface only
- Re-export transitive symbol population — parser complexity not justified

**Technical Debt Incurred:**

- Phases 33 + 35 skipped `/gsd-verify-work` — VERIFICATION.md generated retroactively at milestone close from audit + test files
- Phase 35 plan SUMMARY files use body-level `**Requirements:**` line instead of frontmatter `requirements-completed:` (inconsistent with Phase 33 + 34)
- REQUIREMENTS.md traceability table reconciled at milestone close (was "Pending" for all 30 despite phases complete)
