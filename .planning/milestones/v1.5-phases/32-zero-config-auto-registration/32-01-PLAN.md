<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| repo clone → Claude Code process | `.mcp.json` tells Claude Code to spawn `node dist/mcp-server.js` inside the cloned repo — if the repo is attacker-controlled, Claude Code will execute attacker-controlled code the first time the user opens it |
| filesystem → .gitignore | A mis-scoped glob could silently drop the committed `.mcp.json` from git, breaking the ZERO-01 promise |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-01 | Tampering | `.mcp.json` args field | accept | Relative path `dist/mcp-server.js` points inside the repo; any tampering is already covered by the general "don't open untrusted repos in Claude Code" threat model. No new attack surface beyond what Claude Code itself introduces. |
| T-32-02 | Information Disclosure | `.mcp.json` committed to git | mitigate | File contains no secrets, tokens, or user-specific paths (only relative `dist/mcp-server.js`). Explicitly exclude `--base-dir` and any absolute path per D-03/D-01. |
| T-32-03 | Denial of Service | `.gitignore` accidentally ignores `.mcp.json` | mitigate | Task 2 `git check-ignore -q .mcp.json; [ $? -eq 1 ]` verification asserts the file is not ignored before the plan is considered done. |
| T-32-04 | Elevation of Privilege | Claude Code auto-spawn of MCP server | accept | Claude Code requires user confirmation on first MCP server load per MCP client design; outside Phase 32's scope. |
</threat_model>
