Replace the current `shutdown()` method with:

```typescript
/**
 * Graceful shutdown sequence (BRKR-03, D-01):
 * 1. Stop worker (prevents new job pickup).
 * 2. Await current job completion OR drain timeout — whichever comes first.
 * 3. Destroy all client connections.
 * 4. Close server (stop accepting new connections).
 */
async shutdown(drainTimeoutMs: number = 15_000): Promise<void> {
  // 1. Stop worker — prevents new job pickup
  this.worker.stop();

  // 2. Await current job completion with drain timeout (D-01)
  const pending = this.worker.getCurrentJobPromise();
  if (pending) {
    log(`Waiting for current job to finish (drain timeout: ${drainTimeoutMs}ms)...`);
    const drainTimeout = new Promise<void>(resolve => {
      const t = setTimeout(resolve, drainTimeoutMs);
      t.unref(); // Don't prevent process exit if job finishes first
    });
    await Promise.race([pending, drainTimeout]);
  }

  // 3. Destroy all client connections
  for (const socket of this.connections) {
    socket.destroy();
  }
  this.connections.clear();

  // 4. Close server — stop accepting new connections
  await new Promise<void>((resolve) => this.server.close(() => resolve()));

  log('Broker server shut down');
}
```

Key details:
- The `drainTimeoutMs` parameter has a default of `15_000` (15s, within D-01's 10-30s range) so the signature is backward-compatible.
- The `.unref()` call on the drain timeout timer follows the established pattern from `worker.ts` `scheduleNext()` — prevents the timer from keeping the process alive if the job finishes before timeout.
- Per D-02, queued (not-yet-started) jobs are dropped silently — `this.worker.stop()` prevents new pickup, and the queue is abandoned when the server closes. No explicit queue drain needed.
- The `Promise.race` pattern is the same one used in `worker.ts` `runJobWithTimeout()` for per-job timeout enforcement.

```typescript
/**
 * Polls for the broker socket file to appear after spawning.
 * Returns true when the socket exists, false if the deadline expires.
 * Uses async sleep between checks to avoid blocking the event loop (Pitfall 4).
 */
async function waitForSocket(
  sockPath: string,
  pollIntervalMs: number,
  maxWaitMs: number,
): Promise<boolean> {
  const deadline = Date.now() + maxWaitMs;
  while (Date.now() < deadline) {
    if (existsSync(sockPath)) return true;
    await new Promise<void>(r => setTimeout(r, pollIntervalMs));
  }
  return existsSync(sockPath); // one final check at deadline
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Broker socket file | Client polls for socket existence to determine readiness — a race condition between socket creation and broker listen could cause premature connection |
| Drain timeout | A malicious or stuck LLM job could exhaust the drain timeout, but cleanup proceeds regardless |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-05 | Denial of Service | Drain timeout exhaustion | accept | Drain timeout (15s) bounds the worst case. After timeout, shutdown proceeds with forced cleanup. Stuck job is abandoned. |
| T-29-06 | Denial of Service | Socket poll false readiness | accept | Poll checks socket file existence, not connectivity. `attemptConnect()` immediately follows and handles connection failures with reconnect timer. No additional mitigation needed. |
| T-29-07 | Tampering | Fake socket file during poll | accept | Local attacker with ~/.filescope/ write access already has full control. Poll detecting a fake socket leads to connection error, which triggers reconnect — self-healing. |
</threat_model>
