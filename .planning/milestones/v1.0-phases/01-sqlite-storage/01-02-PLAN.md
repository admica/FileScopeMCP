    - Test: runMigrationIfNeeded() with no JSON files does nothing (returns cleanly)
    - Test: runMigrationIfNeeded() with existing .filescope.db skips migration (DB already exists)
    - Test: runMigrationIfNeeded() with a JSON file but no DB triggers migration
    - Test: Migration handles empty children arrays and missing optional fields (importance=undefined, summary=undefined)
  </behavior>
  <action>
    Create `src/migrate/json-to-sqlite.ts` following RESEARCH.md Pattern 4 with adjustments:

    1. `collectNodes(node: FileNode, out: FileNode[]): void` — recursive helper that flattens the nested FileNode tree into a flat array. Walks `node.children` recursively.

    2. `migrateJsonToSQLite(jsonPath: string, dbPath: string): void` — the core migration function:
       - Read JSON file synchronously with `fs.readFileSync` (migration is a startup operation, sync is fine with better-sqlite3).
       - Parse as `FileTreeStorage` (the existing JSON format wraps config + fileTree).
       - Call `collectNodes()` to flatten the tree.
       - Log count: `log('Migration: migrating N files from JSON to SQLite')`.
       - Open the database via `openDatabase(dbPath)` (this also runs Drizzle migrations to create tables).
       - Use a Drizzle transaction (per locked decision: all-or-nothing):
         - For each node: call `upsertFile(node)` to insert into files table.
         - For each node with `dependencies` or `packageDependencies`: call `setDependencies(node.path, node.dependencies ?? [], node.packageDependencies ?? [])`.
         - CRITICAL per RESEARCH.md Pitfall 3: Do NOT insert `node.dependents` as rows. Dependents are derived by querying `file_dependencies WHERE target_path = ?`.
       - After successful transaction: rename JSON to `.bak` via `fs.renameSync(jsonPath, jsonPath + '.bak')`.
       - Log completion: `log('Migration: complete. JSON backed up to ${jsonPath}.bak')`.
       - On transaction failure: catch error, log it via `log()`, do NOT rename JSON, re-throw so caller knows migration failed.

    3. `runMigrationIfNeeded(projectRoot: string): void` — entry point called from server init:
       - Check if `.filescope.db` already exists at `path.join(projectRoot, '.filescope.db')`. If yes, return (DB exists, no migration needed).
       - Scan projectRoot for files matching `FileScopeMCP-tree*.json` pattern (the existing naming convention from `storage-utils.ts`).
       - If no JSON files found, return (fresh install, no migration needed).
       - For each JSON file found, call `migrateJsonToSQLite(jsonPath, dbPath)` wrapped in try/catch. On failure, log error but do NOT re-throw — server falls back to JSON gracefully (per locked decision).
       - Per RESEARCH.md code example for migration detection.

    IMPORTANT implementation notes:
    - Use `log()` from `../logger.js` for all migration feedback (per locked decision: stderr, not console.log).
    - The transaction should use the raw better-sqlite3 transaction via `getDb().transaction()` for atomicity.
    - Handle edge cases: FileNode with no children (leaf file), FileNode with importance=undefined (default to 0), FileNode with no summary (null).

    Update `package.json` build script to include `src/migrate/json-to-sqlite.ts` in esbuild entry points.
