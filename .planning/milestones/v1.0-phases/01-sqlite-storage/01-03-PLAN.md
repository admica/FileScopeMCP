  <name>Task 1: Rewire server init, storage-utils, file-utils, and global-state for SQLite</name>
  <files>src/mcp-server.ts, src/storage-utils.ts, src/file-utils.ts, src/global-state.ts</files>
  <action>
    Rewire the infrastructure layer: server initialization, storage functions, file mutation functions, and global state. This task does NOT touch tool handlers — those are Task 2.

    **Step 1: Rewire `initializeProject()` in mcp-server.ts**
    - After `setProjectRoot(projectRoot)` and `process.chdir(projectRoot)`, add:
      1. Call `runMigrationIfNeeded(projectRoot)` — handles JSON-to-SQLite migration transparently.
      2. Call `openDatabase(path.join(projectRoot, '.filescope.db'))` — opens (or creates) the SQLite DB with WAL pragmas and migrations.
    - If switching projects (DB already open), call `closeDatabase()` BEFORE opening the new one (per RESEARCH.md Open Question 2).
    - Remove the JSON-based `FileTreeConfig.filename` usage for persistence. The config is still needed for `baseDirectory` and `projectRoot` but NOT for JSON file I/O.

    **Step 2: Rewire `buildFileTree()` in mcp-server.ts**
    - Currently: loads cached JSON, checks freshness, falls back to full scan, saves to JSON.
    - New behavior: check if SQLite has data (via `getAllFiles().length > 0`). If yes, use DB as cache (validate freshness via same spot-check sampling on mtime). If no data or stale, do full scan via `scanDirectory()`, then bulk insert via `upsertFile()` for each node and `setDependencies()` for each file's imports.
    - Remove: `loadFileTree()` call, `saveFileTree()` call. The module-level `fileTree` variable should be populated from DB (reconstruct root FileNode for backward compat with tool handlers that still reference it).
    - Keep: `scanDirectory()`, `buildDependentMap()`, `calculateImportance()` — these still work on in-memory FileNode trees during the scan phase. After scan completes, write results to DB.

    **Step 3: Rewire `handleFileEvent()` in mcp-server.ts**
    - Currently: mutates in-memory `fileTree`, then `saveFileTree()`.
    - New: mutate in-memory tree (for consistency during the operation), then write changes to DB via repository functions. Replace `saveFileTree(currentConfig!, fileTree!)` with individual `upsertFile()`, `deleteFile()`, `setDependencies()` calls as appropriate.
    - For add: After `addFileNode()`, call `upsertFile(newNode)` and `setDependencies(newNode.path, ...)`.
    - For change: After `updateFileNodeOnChange()`, call `upsertFile(updatedNode)` and `setDependencies(...)`.
    - For unlink: After `removeFileNode()`, call `deleteFile(removedPath)`.

    **Step 4: Rewire `integrityCheck`/`startIntegritySweep()` in mcp-server.ts**
    - Currently: operates on in-memory tree, saves to JSON.
    - New: load current file list from DB via `getAllFiles()`, run integrity check against disk, write fixes back via repository.

    **Step 5: Rewire `storage-utils.ts`**
    - `saveFileTree()`: Rewrite to bulk upsert all nodes to DB instead of JSON write. Keep the function signature for now (callers still call it) but internally use repository.
    - `loadFileTree()`: Rewrite to reconstruct `FileTreeStorage` from DB. Build root FileNode with children from `getChildren()` recursively. Keep function signature.
    - `updateFileNode()`: Delegate to `getFile()` + merge updates + `upsertFile()`. Keep function signature.
    - `getFileNode()`: Delegate to `getFile()` from repository. Keep function signature.
    - `clearTreeCache()`: No-op (no more in-memory cache). Keep function for backward compat.
    - **Remove** the `loadedTrees` Map entirely (per RESEARCH.md Pitfall 5 and locked decision to use SQLite as source of truth).

    **Step 6: Update file-utils.ts mutation functions**
    - `addFileNode()`: After adding to tree, also `upsertFile()` to DB.
    - `removeFileNode()`: After removing from tree, also `deleteFile()` from DB.
    - `updateFileNodeOnChange()`: After updating in tree, also `upsertFile()` to DB.
    - These functions still operate on the in-memory tree for the duration of the operation, but persist to DB. Over time (Phase 2+), the in-memory tree may be removed entirely.

    **Step 7: Wire database lifecycle in global-state.ts (if needed)**
    - If `global-state.ts` needs to know about the DB path, add a `getDbPath(): string` helper that returns `path.join(getProjectRoot(), '.filescope.db')`.
